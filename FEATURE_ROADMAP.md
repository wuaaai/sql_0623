# Text-to-SQL 产品功能路线图

## 产品定位

面向财政预算分析人员的自然语言数据库查询助手。用户用口语化业务术语描述需求，系统自动完成口语→数据库表/列/值的映射，生成正确 SQL，返回结果。

**数据环境**: 达梦 · Schema: RDYS_PUBLIC_TBS · 47张表 · 四大预算类型
**当前阶段**: 阶段3完成，进入阶段4

---

## 阶段边界原则

每个阶段交付一组互不重叠的能力：

```
阶段1: 能看数据 — 浏览、探索、单表查询
阶段2: 能筛数据 — 行级条件：地区/时间/项目/同比
阶段3: 能汇数据 — 组级聚合：分组/排名/趋势/统计
阶段4: 能跨表分析 — 多表对比/合并/全景
阶段5: 能深度分析 — 子查询/占比/环比/异常检测
阶段6: 能记对话 — 跨会话上下文持久化、查询历史、个性化默认值
阶段7: 能自修正 — SQL错误自动诊断修复
```

### 阶段1: 数据探索 ✅
**一句话**: 连接数据库，看表、搜表、查看表结构和样例数据。

**原子工具**: connect_db, list_tables, search_schema, describe_table, run_sql

---

### 阶段2: 精准筛选 ✅
**一句话**: 用户用口语描述筛选条件，系统生成行级 WHERE 查询。

**能力边界**: 预算类型映射、地区/时间/项目筛选、智能列选择、时间解析、同比列选择、追问限制（最多2次）、上下文推断

**原子工具**: + resolve_time, suggest_columns

---

### 阶段3: 聚合分析 ✅
**一句话**: 跨行汇总统计——分组排名、时间趋势、统计指标。

**能力边界**: GROUP BY 聚合、排名 TOP N、时间趋势、AVG/SUM/COUNT、多维分组

**原子工具**: + run_aggregation

**关键发现**:
- 数据库有大量预聚合列（累计值、同比率），多数统计问题无需 GROUP BY
- 排名查询直接 ORDER BY 金额列即可（每行已是一个地区的数据）
- 真正需要 GROUP BY 的是跨月汇总各市、跨项目汇总等场景

---

### 阶段4: 跨表分析 ✅
**一句话**: 跨预算类型对比分析——并行查询+LLM综合对比、同构表 UNION ALL 合并、字典表 JOIN 关联、表关系智能发现。

**原子工具**: + find_relations, union_query（共10个工具）

**技能**: cross_table_analysis.md（4种跨表策略）

---

### 阶段5: 深度洞察 ✅
**一句话**: 子查询、窗口函数、异常检测——从"看懂数据"到"发现洞察"。

**原子工具**: + run_subquery, calc_ratio, detect_anomalies（共13个工具）

**技能**: advanced_analysis.md（子查询/占比/环比LAG/异常检测4种模式）

---

### 阶段6: 会话记忆 📋

**产品价值**: 用户关闭浏览器后重新打开，Agent 记得上次查过什么。从"每次重新开始"到"接着上次继续"。

**核心区分**: 阶段1-5解决"当下怎么查"，阶段6解决"下次怎么接上"。不是新的查询能力，而是查询体验的跃迁。

#### 真实用户场景

| 用户场景 | 现在的痛点 | 阶段6的体验 |
|---------|-----------|-----------|
| 昨天分析了河北省一般公共预算，今天打开浏览器想看社保 | 重新输入"河北省"、选预算类型、选时间 | 直接说"社保的呢"，Agent 复用昨天的上下文 |
| 每天早上领导要看收入完成情况 | 每次都要完整描述查询条件 | 一键"继续上次分析"，自动加载默认条件 |
| 上周查过一个异常值，这周想看看变化 | 不记得当时用的什么条件 | 查询历史列表，点击重新执行 |
| 新同事接手工作，想看看之前怎么查的 | 无从下手 | 浏览历史对话，理解查询模式 |

#### 为什么前面的阶段搞不定

阶段2的上下文推断只在**同一会话内**有效——刷新页面就丢失。阶段6要持久化到磁盘，跨浏览器会话可用。

#### 功能6.1: 会话自动保存与恢复

每次对话自动保存到服务端 JSON 文件。下次打开浏览器时自动恢复最近一次会话的上下文。

**上下文记忆内容**:
- 最后使用的预算类型（一般公共预算/社保/国资/政府性基金）
- 最后关注的地区
- 最后使用的时间范围
- 最后操作的表名

**用户感知**:
- 页面顶部显示："继续上次分析 — 河北省 · 一般公共预算 · 2026年4月"
- 直接输入问题，Agent 自动带入上下文
- 也可以"开始新对话"清空上下文

#### 功能6.2: 查询历史

| 用户操作 | 系统行为 |
|---------|---------|
| 查看历史查询列表 | 展示最近20条查询（问题+生成的SQL+时间戳） |
| 点击历史查询 | 重新展示当时的结果 |
| 基于历史查询修改 | 复制历史查询的上下文，用户改一个条件 |

**实现方式**:
- 服务端每个会话保存为 `sessions/{session_id}.json`
- 前端 localStorage 记录 `session_id`
- 服务启动时自动加载历史会话列表

#### 功能6.3: 个性化默认值

| 用户设置 | 效果 |
|---------|------|
| 默认地区=河北省 | 所有查询默认 WHERE RG_NAME='河北省' |
| 默认预算=一般公共预算 | "收入多少"自动理解为"一般公共预算收入" |
| 默认时间=最新 | resolve_time("") 自动生效 |

保存在浏览器 localStorage，无需服务端配置。

#### 新增原子工具: 无

阶段6主要是基础设施（JSON文件读写、前端localStorage），不需要新的数据库原子工具。

**新增技能**: `session_memory.md` — 会话记忆策略（何时复用上下文、何时重置、历史查询复用模板）

#### 验收标准
- 关闭浏览器重新打开 → 看到"继续上次分析"提示
- 说"社保的呢" → Agent 复用上次的地区和时间
- 查询历史列表 → 能看到之前的问题和生成的SQL
- "开始新对话" → 上下文完全清空

### 阶段7: 智能容错 📋

**产品价值**: SQL报错时Agent自动修正，用户看不到技术错误。从"报错了找管理员"到"自动修复无感知"。

**核心场景**:

| 错误 | 现在的表现 | 阶段7的表现 |
|------|---------|-----------|
| 列名写错 | "column X not found"给用户 | describe_table重新确认→自动修正列名→重试 |
| 表名找不到 | "table X not found" | search_schema重新定位→自动替换 |
| ROWNUM位置错误 | SQL语法错误 | 自动调整子查询嵌套 |
| 查询超时 | 卡住不动 | 自动加ROWNUM限制重试 |
| 数据类型不匹配 | "type mismatch" | 自动加类型转换函数 |

**实现方式**: handler层捕获SQL错误→分析错误信息→自动修正→透明重试（用户只看到最终成功结果）

**原子工具**: 无需新增，修改现有handler的错误处理逻辑。

## 质量基线（所有阶段通用）

1. **不编造**: 所有数据来自工具返回，禁止编造表名/列名/数值
2. **工具优先**: 用户消息注入强制指令，确保每次查询都走工具链
3. **达梦语法正确**: ROWNUM（非 LIMIT），||（非 CONCAT），ORDER BY + ROWNUM 嵌套子查询
4. **不 SELECT ***: 按意图选列
5. **流式输出**: LLM 响应逐 token 流式展示，完成后渲染 markdown
6. **口语化优先**: 所有用户界面和测试问题使用真实口语，不暴露英文表名/列名
