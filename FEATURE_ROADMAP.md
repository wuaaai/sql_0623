# Text-to-SQL 产品功能路线图

## 产品定位

面向财政预算分析人员的自然语言数据库查询助手。用户用口语化业务术语描述需求，系统自动完成口语→数据库表/列/值的映射，生成正确 SQL，返回结果。

**数据环境**: 达梦 · Schema: RDYS_PUBLIC_TBS · 47张表 · 四大预算类型
**当前阶段**: 阶段3完成，进入阶段4

---

## 阶段边界原则

每个阶段交付一组互不重叠的能力：

```
阶段1: 能看数据 — 浏览、探索、单表查询
阶段2: 能筛数据 — 行级条件：地区/时间/项目/同比
阶段3: 能汇数据 — 组级聚合：分组/排名/趋势/统计
阶段4: 能跨表分析 — 多表对比/合并/全景
阶段5: 能深度分析 — 窗口函数/子查询/异常检测
阶段6: 能记对话 — 跨会话上下文持久化
阶段7: 能自修正 — 错误诊断/自动修复/降级
```

### 阶段1: 数据探索 ✅
**一句话**: 连接数据库，看表、搜表、查看表结构和样例数据。

**原子工具**: connect_db, list_tables, search_schema, describe_table, run_sql

---

### 阶段2: 精准筛选 ✅
**一句话**: 用户用口语描述筛选条件，系统生成行级 WHERE 查询。

**能力边界**: 预算类型映射、地区/时间/项目筛选、智能列选择、时间解析、同比列选择、追问限制（最多2次）、上下文推断

**原子工具**: + resolve_time, suggest_columns

---

### 阶段3: 聚合分析 ✅
**一句话**: 跨行汇总统计——分组排名、时间趋势、统计指标。

**能力边界**: GROUP BY 聚合、排名 TOP N、时间趋势、AVG/SUM/COUNT、多维分组

**原子工具**: + run_aggregation

**关键发现**:
- 数据库有大量预聚合列（累计值、同比率），多数统计问题无需 GROUP BY
- 排名查询直接 ORDER BY 金额列即可（每行已是一个地区的数据）
- 真正需要 GROUP BY 的是跨月汇总各市、跨项目汇总等场景

---

### 阶段4: 跨表分析 ✅
**一句话**: 跨预算类型对比分析——并行查询+LLM综合对比、同构表 UNION ALL 合并、字典表 JOIN 关联、表关系智能发现。

**原子工具**: + find_relations, union_query（共10个工具）

**技能**: cross_table_analysis.md（4种跨表策略）

---

### 阶段5: 深度洞察 📋

**产品价值**: 回答领导的高阶分析问题——"哪个异常？占比多少？比平均高还是低？这个月和上个月比怎么样？" 从"看懂数据"到"发现洞察"。

**核心区分**: 阶段1-4回答了"是什么"（数据多少、排名第几、对比如何），阶段5回答"为什么"和"有多特别"（异常吗、占比合理吗、趋势健康吗）。

#### 为什么阶段4搞不定

阶段4处理"多张表之间怎么关联和对比"。阶段5处理"单张表内的深度计算"——窗口函数、子查询、标准差、百分位。这些是 SQL 的高级特性，LLM 不会主动写，需要专门的工具和技能引导。

#### 真实用户场景

| 用户怎么说 | 真实需求 | SQL 技术 |
|-----------|---------|---------|
| "哪些地市收入超过全省平均水平" | 与平均值对比 | 子查询 `WHERE BYS_JE > (SELECT AVG(BYS_JE) ...)` |
| "各地市收入占全省的比例" | 占比计算 | `BYS_JE / SUM(BYS_JE) OVER()` |
| "这个月和上个月比增长了多少" | 环比 | `LAG(BYS_JE, 1) OVER (PARTITION BY RG_NAME ORDER BY YEAR_MONTH)` |
| "有没有收入特别异常的" | 异常检测 | `ABS(BYS_JE - AVG) > 2 * STDDEV` |
| "收入排名前20%的地市" | 百分位排名 | `PERCENT_RANK() OVER (ORDER BY BYS_JE DESC)` |
| "按收入分档：高/中/低" | 分档 | `NTILE(3) OVER (ORDER BY BYS_JE DESC)` |
| "各地市收入排名及与上一名的差距" | 排名+差值 | `RANK() + LAG` 组合 |
| "全年收入最高的三个月" | TOP N 排序 | `ROW_NUMBER() OVER (ORDER BY BYS_JE DESC)` |

#### 功能5.1: 子查询——"比平均高/低"

用户问"哪些地市收入超过全省平均"时：
1. 先算全省平均: `SELECT AVG(BYS_JE) FROM table WHERE ...`
2. 再查超过平均的: `SELECT RG_NAME, BYS_JE FROM table WHERE BYS_JE > (子查询)`

| 用户输入 | 系统行为 |
|---------|---------|
| "收入超过全省平均的地市" | 两阶段：先算平均 → 再筛选超平均 |
| "支出低于平均水平的地市" | 同上，反向筛选 |
| "比石家庄收入高的地市" | `WHERE BYS_JE > (SELECT BYS_JE WHERE RG_NAME='石家庄市')` |

#### 功能5.2: 占比分析——"占多少"

| 用户输入 | 系统行为 |
|---------|---------|
| "各地市收入占全省的比例" | `BYS_JE / SUM(BYS_JE) OVER() * 100` |
| "税收收入占一般公共预算收入的比重" | 税收行金额 / 合计行金额 |
| "前3名地市收入占比合计" | TOP 3 SUM / 全省 SUM |

#### 功能5.3: 环比——"和上个月比"

| 用户输入 | 系统行为 |
|---------|---------|
| "这个月收入比上个月增长了多少" | LAG 窗口函数获取上月值 → 计算差值 |
| "各月环比变化趋势" | 逐月 LAG → 计算环比增长率 |
| "哪些地市本月收入环比下降" | LAG → WHERE 本月 < 上月 |

#### 功能5.4: 异常检测——"有没有不对劲的"

| 用户输入 | 系统行为 |
|---------|---------|
| "有没有收入特别异常的地市" | 计算均值±2倍标准差 → 标记异常值 |
| "同比波动最大的地市" | 计算同比率的方差 → 排序 |
| "收入突然大幅下降的地区" | LAG 对比 → 降幅 > 阈值 |

#### 新增原子工具

| 工具 | 用途 |
|------|------|
| `run_subquery` | 执行"先算A再查B"的两阶段子查询 |
| `detect_anomalies` | 异常检测：均值±N倍标准差，标记离群值 |
| `calc_ratio` | 占比计算：部分/整体，自动处理窗口函数 |

#### 新增技能

`advanced_analysis.md` — 深度分析策略：何时用子查询、何时用窗口函数、环比模板、异常检测阈值。

#### 验收标准
- "超过全省平均的地市" → 子查询正确筛选
- "各地市占比" → 百分比正确，总和=100%
- "环比变化" → LAG 正确计算
- "异常检测" → 标记合理、不误报

### 阶段6: 会话记忆 📋

**产品价值**: 用户关闭浏览器后重新打开，Agent 记住上次查过什么。从"每次重新开始"到"接着上次继续"。

| 用户场景 | 系统行为 |
|---------|---------|
| 昨天查了河北省一般公共预算，今天打开直接问"社保的呢" | 复用昨天的地区上下文 |
| 每周一问"本周收入完成情况" | 记住常用查询模板，一键执行 |
| "默认只看河北省的数据" | 个性化默认值持久化 |

**实现方式**: 服务端会话文件（JSON）+ 前端 localStorage 结合。

---

### 阶段7: 智能容错 📋

**产品价值**: SQL 报错时 Agent 自动诊断原因并修正，用户看不到技术错误信息。

| 错误场景 | 系统行为 |
|---------|---------|
| 列名写错 | describe_table 重新确认 → 自动修正 |
| 表名找不到 | search_schema 重新定位 → 自动替换 |
| ROWNUM 位置错误 | 自动调整为正确的子查询嵌套 |
| 超时 | 自动加 ROWNUM 限制重试 |

## 质量基线（所有阶段通用）

1. **不编造**: 所有数据来自工具返回，禁止编造表名/列名/数值
2. **工具优先**: 用户消息注入强制指令，确保每次查询都走工具链
3. **达梦语法正确**: ROWNUM（非 LIMIT），||（非 CONCAT），ORDER BY + ROWNUM 嵌套子查询
4. **不 SELECT ***: 按意图选列
5. **流式输出**: LLM 响应逐 token 流式展示，完成后渲染 markdown
6. **口语化优先**: 所有用户界面和测试问题使用真实口语，不暴露英文表名/列名
