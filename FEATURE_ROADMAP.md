# Text-to-SQL 产品功能路线图

## 产品定位

面向财政预算分析人员的自然语言数据库查询助手。用户用口语化业务术语描述需求，系统自动完成口语→数据库表/列/值的映射，生成正确 SQL，返回结果。

**数据环境**: 达梦 · Schema: RDYS_PUBLIC_TBS · 47张表 · 四大预算类型

---

## 阶段边界设计原则

每个阶段交付一组**互不重叠**的能力。编号越大的阶段，能力建立在前面所有阶段之上，但不重复前面阶段已交付的功能。

**核心区分逻辑：**

```
阶段1 → 能看到数据（浏览、探索、单表明细）
阶段2 → 能筛选数据（行级条件：地区/时间/项目/同比）  
阶段3 → 能汇总数据（组级聚合：分组/排名/趋势/统计指标）
阶段4 → 能关联数据（跨表操作：JOIN/UNION/同环比）
阶段5 → 能分析数据（高级分析：窗口函数/子查询/异常检测）
阶段6 → 能记住对话（会话管理：跨轮次上下文持久化）
阶段7 → 能自我修正（容错：错误诊断/自动修正/降级处理）
阶段8 → 能写入数据（安全写：确认机制/权限控制/审计）
```

**判断一个问题属于哪个阶段的标准：**

| 用户问法特征 | 所属阶段 | SQL 特征 |
|------------|---------|---------|
| "XX表有哪些数据" / "看看XX表结构" | 阶段1 | SELECT * / DESCRIBE |
| "河北省XX收入" / "2025年1月合计" | 阶段2 | WHERE 条件筛选行 |
| "各市收入排名" / "各月趋势" / "平均" | 阶段3 | GROUP BY + 聚合函数 |
| "一般公共预算和社保对比" | 阶段4 | JOIN / UNION |
| "比全省平均高的地市" / "排名前20%" | 阶段5 | 窗口函数 / 子查询 |
| （连续多轮同一话题对话） | 阶段6 | 上下文记忆复用 |
| SQL报错后自动修正 | 阶段7 | 错误处理 |
| "修改XX数据" | 阶段8 | INSERT/UPDATE/DELETE |

### 阶段1: 数据探索 ✅
**一句话**: 用户能连接数据库，看到有哪些表，查看表结构和样例数据，做最简单的单表浏览。

**能力边界**: 
- ✅ 连接数据库、列出表、搜索表（中文注释匹配）、查看表结构
- ✅ 最简单的 SELECT * 浏览（不加筛选条件或仅 ROWNUM 限制）
- ❌ 不涉及 WHERE 条件筛选（那是阶段2）
- ❌ 不涉及 GROUP BY 聚合（那是阶段3）

**原子工具**: connect_db, list_tables, search_schema, describe_table, run_sql

---

### 阶段2: 精准筛选 ✅
**一句话**: 用户用口语描述筛选条件，系统自动定位表和列，生成行级条件查询。

**能力边界**:
- ✅ 预算类型→表名自动映射（"一般公共预算"→YBGGYS）
- ✅ 行级 WHERE 条件：地区、时间、项目名称
- ✅ 时间智能解析：无时间→最新，相对时间→具体年月
- ✅ 同比列智能选择：用户说"增长"→自动选 BYS_TBE/BYS_TBB
- ✅ 多条件组合：地区+时间+项目+N个条件叠加
- ✅ 智能列选择：按意图推荐列，不 SELECT *
- ✅ 上下文推断：复用上一轮的地区/预算/时间，不问重复问题
- ✅ 追问限制：最多2次，第3次自动用默认值执行
- ❌ 不涉及 GROUP BY 分组（那是阶段3）
- ❌ 不涉及 SUM/AVG/COUNT 聚合（那是阶段3）
- ❌ 不涉及 ORDER BY 排名（简单的 ASC/DESC 展示排序除外）

**重要澄清**——以下问题虽然包含"各市"、"排名"等词，但属于阶段2的行级查询而非阶段3的聚合：
- "各市2025年政府性基金累计收入" → 行级查询，WHERE 排除本级，选 BYLJS_JE 列即可（数据库里每行已经有各市的数据，不需要 SUM）
- "河北省一般公共预算收入合计" → 行级查询，WHERE 地区+项目=合计（合计是 XM_NAME 的一个值，不需要 SUM）

**原子工具**: + resolve_time, suggest_columns

---

### 阶段3: 聚合分析 📋
**一句话**: 用户能做跨行汇总统计——分组、排名、趋势、平均/最高/最低/总和。这是"从看到数据"到"读懂数据"的跃迁。

**产品价值**: 回答领导的决策性问题——"哪个最好？哪个最差？趋势怎么样？总体是多少？"

**能力边界**:
- ✅ GROUP BY 分组聚合（按地区/时间/项目分组，对金额列求和/平均/计数）
- ✅ 排序排名（按聚合结果降序/升序，取 TOP N）
- ✅ 时间趋势（各月收入/支出变化，GROUP BY YEAR_MONTH）
- ✅ 统计指标（AVG/MAX/MIN/SUM/COUNT）
- ✅ 汇总合计（跨时间范围的 SUM）
- ❌ 不涉及跨表操作（那是阶段4）
- ❌ 不涉及窗口函数/子查询（那是阶段5）

**阶段3 vs 阶段2 的清晰边界**:

| 用户问题 | 属于 | 原因 |
|---------|------|------|
| "河北省2025年1月一般公共预算收入合计" | 阶段2 | 行级查询，筛选到具体一行 |
| "各市一般公共预算收入合计" | 阶段3 | 需要 GROUP BY RG_NAME + SUM |
| "石家庄社保基金本月收入" | 阶段2 | 行级查询，筛选到具体一行 |
| "社保基金收入最高的5个地市" | 阶段3 | 需要 GROUP BY + SUM + ORDER BY + ROWNUM |
| "今年各月收入" | 阶段2 | 行级查询，各月数据本身就是多行 |
| "今年各月收入趋势" | 阶段3 | 需要 GROUP BY YEAR_MONTH + SUM（可能有多行同月） |
| "各地市平均收入" | 阶段3 | 需要 AVG 聚合函数 |
| "全省收入汇总" | 阶段3 | 需要 SUM 聚合函数 |

**为什么阶段3需要新能力而阶段2搞不定**:
阶段2生成的行级SQL（如 `SELECT RG_NAME, BYS_JE FROM table WHERE ...`）返回的是每行原始数据。如果同一地区有多行（不同项目、不同月份），用户要"各市汇总"时，LLM 必须在 SQL 层面做 SUM+GROUP BY，而不能拿到明细行后自己加——那样不准确、不完整、违反"数据库做聚合"原则。

**功能详情**:

#### 3.1 分组汇总
用户："各市一般公共预算收入"  
SQL: `SELECT RG_NAME, SUM(BYS_JE) as total FROM table WHERE ... GROUP BY RG_NAME ORDER BY total DESC`

用户："各类税收收入情况"  
SQL: `SELECT XM_NAME, SUM(BYS_JE) as total FROM table WHERE ... GROUP BY XM_NAME`

#### 3.2 排名 TOP N
用户："收入最高的5个地市"  
SQL（达梦）:
```sql
SELECT * FROM (
  SELECT RG_NAME, SUM(BYS_JE) as total
  FROM table WHERE ...
  GROUP BY RG_NAME ORDER BY total DESC
) WHERE ROWNUM <= 5
```

用户："同比增幅最大的10个地区"  
SQL: 同上模式，ORDER BY AVG(BYS_TBB) DESC

#### 3.3 时间趋势
用户："今年各月一般公共预算收入趋势"  
SQL: `SELECT YEAR_MONTH, SUM(BYS_JE) as total FROM table WHERE YEAR_MONTH LIKE '2026%' AND XM_NAME LIKE '%合计%' GROUP BY YEAR_MONTH ORDER BY YEAR_MONTH`

#### 3.4 统计指标
用户："各地市平均收入"  
SQL: `SELECT AVG(total) FROM (SELECT RG_NAME, SUM(BYS_JE) as total FROM table WHERE ... GROUP BY RG_NAME)`

用户："收入最高和最低的地区"  
SQL: 两个子查询分别取 MAX 和 MIN，或者一个排名子查询取 ROWNUM=1 ASC 和 DESC

#### 3.5 汇总合计
用户："2025年全年社保基金总收入"  
SQL: `SELECT SUM(BYS_JE) as total FROM table WHERE YEAR_MONTH LIKE '2025%' AND XM_NAME LIKE '%合计%'`

#### 3.6 组合聚合
用户："各市各月一般公共预算收入"  
SQL: `SELECT RG_NAME, YEAR_MONTH, SUM(BYS_JE) as total FROM table WHERE ... GROUP BY RG_NAME, YEAR_MONTH ORDER BY RG_NAME, YEAR_MONTH`

**验收标准**:
- "各市收入排名" → GROUP BY RG_NAME + SUM + ORDER BY + ROWNUM
- "收入最高的5个地市" → 聚合排名子查询
- "今年各月收入趋势" → GROUP BY YEAR_MONTH + SUM
- "各地市平均收入" → AVG 聚合
- "2025年全年总收入" → SUM 汇总
- 所有聚合 SQL 使用 GROUP BY + 聚合函数，不使用 SELECT DISTINCT 伪装聚合

**需要的原子工具变更**:
- 新增 `run_aggregation` — 接收分组列、聚合列、聚合函数、排序和限制，生成正确的聚合 SQL 模板。解决 LLM 不擅长写 GROUP BY 的问题。
- 或增强 `suggest_columns` — 当意图为"排名/汇总/趋势"时，返回聚合 SQL 模板而非仅列名。

### 阶段4: 跨表关联 📋
**一句话**: 用户能跨多张表做查询——同构表合并、预算全景对比、基础表关联。

**能力边界**:
- ✅ 同构表 UNION ALL（如一般公共预算收入 + 社保基金收入）
- ✅ 预算全景对比（四大预算的收入/支出/完成率并排展示）
- ✅ 基础表 JOIN（支出分类 × 地区划分的对照）
- ❌ 不涉及窗口函数/子查询（那是阶段5）

**为什么阶段2/3搞不定**: 阶段2/3只操作单张表。跨表查询需要多张表的列对齐、UNION 或 JOIN 语法。

---

### 阶段5: 深度洞察 📋
**一句话**: 用户能做高级分析——异常检测、占比分布、同环比、预测对比。

**能力边界**:
- ✅ 窗口函数（ROW_NUMBER, RANK, LAG/LEAD）
- ✅ 子查询（比全省平均高的地区）
- ✅ 占比计算（各地市占全省比例）
- ✅ 异常检测（偏离均值超过2倍标准差的地区）
- ✅ 同环比计算（本月 vs 上月）
- ❌ 不涉及跨会话记忆（那是阶段6）

**为什么阶段3搞不定**: 阶段3只有基础聚合。窗口函数和子查询是更高级的 SQL 能力，需要额外的语法知识。

---

### 阶段6: 会话记忆 📋
**一句话**: 用户关闭浏览器后重新打开，Agent 记得上次聊过什么、查过哪些表、关注哪个地区。

**能力边界**:
- ✅ 跨会话上下文持久化
- ✅ 常用查询模板（"每天早上帮我看一下收入"）
- ✅ 个性化默认值（"我只看河北省的"）
- ❌ 不涉及错误处理（那是阶段7）

---

### 阶段7: 智能容错 📋
**一句话**: SQL 报错时 Agent 自动诊断原因并修正，用户看不到技术错误。

**能力边界**:
- ✅ SQL 语法错误→根据数据库报错信息修正
- ✅ 列名不存在→describe_table 重新确认
- ✅ 表名错误→search_schema 重新定位
- ✅ 超时→自动加 ROWNUM 限制重试
- ❌ 不涉及写操作（那是阶段8）

---

### 阶段8: 安全写入 📋
**一句话**: 授权用户通过自然语言修改数据，所有写操作经过确认和安全校验。

**能力边界**:
- ✅ INSERT/UPDATE 操作（需用户确认）
- ✅ 批量操作安全拦截（无 WHERE 的 DELETE 拒绝执行）
- ✅ 操作审计日志
- ✅ 回滚机制

---

## 质量基线（所有阶段通用）

1. **不编造**: 所有数据必须来自工具返回，禁止编造表名/列名/数值
2. **不翻译列名**: 表名列名使用原始英文，可在旁边加中文说明
3. **达梦语法正确**: ROWNUM（非 LIMIT），||（非 CONCAT），ORDER BY + ROWNUM 嵌套子查询
4. **安全第一**: 只允许 SELECT，写操作需确认
5. **不 SELECT ***: 除非用户明确要求，按意图选列
6. **数据库做聚合**: 统计查询必须用 GROUP BY + 聚合函数，不能查明细行让 LLM 自己算
