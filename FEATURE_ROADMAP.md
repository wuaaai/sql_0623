# Text-to-SQL 产品功能路线图

## 产品定位

面向财政预算分析人员的自然语言数据库查询助手。用户用口语化业务术语描述需求，系统自动完成口语→数据库表/列/值的映射，生成正确 SQL，返回结果。

**数据环境**: 达梦 · Schema: RDYS_PUBLIC_TBS · 47张表 · 四大预算类型
**当前阶段**: 阶段3完成，进入阶段4

---

## 阶段边界原则

每个阶段交付一组互不重叠的能力：

```
阶段1: 能看数据 — 浏览、探索、单表查询
阶段2: 能筛数据 — 行级条件：地区/时间/项目/同比
阶段3: 能汇数据 — 组级聚合：分组/排名/趋势/统计
阶段4: 能跨表分析 — 多表对比/合并/全景
阶段5: 能深度分析 — 窗口函数/子查询/异常检测
阶段6: 能记对话 — 跨会话上下文持久化
阶段7: 能自修正 — 错误诊断/自动修复/降级
```

### 阶段1: 数据探索 ✅
**一句话**: 连接数据库，看表、搜表、查看表结构和样例数据。

**原子工具**: connect_db, list_tables, search_schema, describe_table, run_sql

---

### 阶段2: 精准筛选 ✅
**一句话**: 用户用口语描述筛选条件，系统生成行级 WHERE 查询。

**能力边界**: 预算类型映射、地区/时间/项目筛选、智能列选择、时间解析、同比列选择、追问限制（最多2次）、上下文推断

**原子工具**: + resolve_time, suggest_columns

---

### 阶段3: 聚合分析 ✅
**一句话**: 跨行汇总统计——分组排名、时间趋势、统计指标。

**能力边界**: GROUP BY 聚合、排名 TOP N、时间趋势、AVG/SUM/COUNT、多维分组

**原子工具**: + run_aggregation

**关键发现**:
- 数据库有大量预聚合列（累计值、同比率），多数统计问题无需 GROUP BY
- 排名查询直接 ORDER BY 金额列即可（每行已是一个地区的数据）
- 真正需要 GROUP BY 的是跨月汇总各市、跨项目汇总等场景

---

### 阶段4: 跨表分析 📋

**产品价值**: 用户能跨预算类型对比分析，看到"全景图"。这是从"看一棵树"到"看整片森林"的跃迁。

**真实用户场景**:

| 用户怎么说 | 真实需求 |
|-----------|---------|
| "一般公共预算和社保基金收入对比" | 分别查两张收入表，并列展示对比 |
| "四大预算收入支出全景" | 查四类预算的合计，汇总成一张总表 |
| "预算执行表和支出分类表怎么关联的" | 理解表之间的关联键（如 EXP_FUNC_CODE） |
| "各市一般公共预算和社保基金的合计收入排名" | 分别查两类预算各市收入，加起来排名 |

**数据库实际情况**:
- 四大预算收入表共享 18 个公共列（RG_CODE, RG_NAME, YEAR_MONTH, DEPT_CODE 等）
- 无显式外键约束，但存在隐式关联键（科目编码、地区编码、年月）
- 字典表（RDYS_BAS_*）可通过编码列关联到事实表（RDYS_LD_*）

**阶段4 能力边界**:
- ✅ 多表分别查询 + LLM 综合对比（主要方式）
- ✅ 同构表 UNION ALL 合并查询（结构相同的表）
- ✅ 预算全景对比（四大预算收入/支出/完成率并排）
- ✅ 表关系发现（自动识别公共列，建议关联方式）
- ❌ 不涉及窗口函数/子查询（那是阶段5）

#### 功能4.1: 跨预算对比

用户问"一般公共预算和社保基金收入哪个多"时：
1. 分别查两张表的合计收入
2. LLM 对比结果，给出结论

不需要真正写 JOIN SQL。Agent 内部并行或串行查两张表，然后综合对比。

| 用户输入 | 系统行为 |
|---------|---------|
| "一般公共预算和社保收入对比" | 查 YBGGYS 收入 + SHBXJJ 收入 → 对比展示 |
| "四大预算收入排名" | 查四类预算收入 → 汇总排名 |
| "哪个预算类型支出最大" | 查四类支出 → 找出最大值 |

#### 功能4.2: 同构表合并

四类预算的收入表结构相同（共享18列），可以 UNION ALL 合并后统一聚合。

| 用户输入 | 系统行为 |
|---------|---------|
| "四大预算各市收入总排名" | UNION ALL 四张表 → GROUP BY RG_NAME → SUM → ORDER BY |
| "各月四大预算收入合计" | UNION ALL → GROUP BY YEAR_MONTH → SUM |

#### 功能4.3: 字典表关联

基础字典表（RDYS_BAS_EXPFUNC 支出分类、RDYS_BAS_MOFDIV 地区划分）通过编码列关联到执行表。

| 用户输入 | 系统行为 |
|---------|---------|
| "按支出分类汇总一般公共预算支出" | JOIN RDYS_BAS_EXPFUNC ON EXP_FUNC_CODE → GROUP BY EXP_FUNC_NAME |
| "各地区各支出分类的预算完成情况" | JOIN 字典表 + 执行表 → 多维分组 |

#### 功能4.4: 表关系智能发现

用户问"这两张表怎么关联"时，自动分析两表的公共列，建议关联方式。

**新增原子工具**: `find_relations(table_a, table_b)` — 分析两表的公共列和外键关系，返回建议的 JOIN 条件。

**新增技能**: `cross_table_analysis.md` — 跨表查询策略（何时并行查询、何时 UNION、何时 JOIN）

**验收标准**:
- "一般公共预算和社保收入对比" → 正确展示两表数据对比
- "四大预算收入排名" → 聚合四类预算的合计收入
- "支出分类表怎么关联到执行表" → 发现公共编码列，建议 JOIN 条件

---

### 阶段5: 深度洞察 📋

**产品价值**: 窗口函数、子查询、异常检测、占比分布——从"看懂数据"到"发现洞察"。

**能力边界**:
- ✅ 窗口函数（ROW_NUMBER, RANK, LAG/LEAD）
- ✅ 子查询（"比平均水平高的地区"）
- ✅ 占比计算（"各地市占全省比例"）
- ✅ 异常检测（"偏离均值2倍标准差"）
- ✅ 环比计算（"本月 vs 上月"）

**为什么阶段4搞不定**: 阶段4处理"多张表"的问题，阶段5处理"同一张表内的高级计算"。

---

### 阶段6: 会话记忆 📋

**产品价值**: 用户关闭浏览器后重新打开，Agent 记住上次查过什么、关注哪个地区、常用哪种预算类型。

**能力边界**:
- ✅ 会话历史持久化（浏览器 localStorage 或服务端存储）
- ✅ 常用查询模板复用
- ✅ 个性化默认值（"默认查河北省的数据"）

---

### 阶段7: 智能容错 📋

**产品价值**: SQL 报错时 Agent 自动诊断原因并修正，用户看不到技术错误信息。

**能力边界**:
- ✅ SQL 语法错误 → 根据数据库报错信息修正
- ✅ 列名不存在 → describe_table 重新确认
- ✅ 表名错误 → search_schema 重新定位
- ✅ 超时 → 自动加 ROWNUM 限制重试

---

## 质量基线（所有阶段通用）

1. **不编造**: 所有数据来自工具返回，禁止编造表名/列名/数值
2. **工具优先**: 用户消息注入强制指令，确保每次查询都走工具链
3. **达梦语法正确**: ROWNUM（非 LIMIT），||（非 CONCAT），ORDER BY + ROWNUM 嵌套子查询
4. **不 SELECT ***: 按意图选列
5. **流式输出**: LLM 响应逐 token 流式展示，完成后渲染 markdown
6. **口语化优先**: 所有用户界面和测试问题使用真实口语，不暴露英文表名/列名
