# Text-to-SQL 产品功能路线图

## 产品定位

面向财政预算分析人员的自然语言数据库查询助手。用户用口语化业务术语描述需求，系统自动完成口语→数据库表/列/值的映射，生成正确 SQL，返回结果。

**数据环境**: 达梦 · Schema: RDYS_PUBLIC_TBS · 47张表 · 四大预算类型
**当前阶段**: 阶段3完成，进入阶段4

---

## 阶段边界原则

每个阶段交付一组互不重叠的能力：

```
阶段1: 能看数据 — 浏览、探索、单表查询
阶段2: 能筛数据 — 行级条件：地区/时间/项目/同比
阶段3: 能汇数据 — 组级聚合：分组/排名/趋势/统计
阶段4: 能跨表分析 — 多表对比/合并/全景
阶段5: 能深度分析 — 子查询/占比/环比/异常检测
阶段6: 能记对话 — 跨会话上下文持久化、查询历史、个性化默认值
阶段7: 能自修正 — SQL错误自动诊断修复
```

### 阶段1: 数据探索 ✅
**一句话**: 连接数据库，看表、搜表、查看表结构和样例数据。

**原子工具**: connect_db, list_tables, search_schema, describe_table, run_sql

---

### 阶段2: 精准筛选 ✅
**一句话**: 用户用口语描述筛选条件，系统生成行级 WHERE 查询。

**能力边界**: 预算类型映射、地区/时间/项目筛选、智能列选择、时间解析、同比列选择、追问限制（最多2次）、上下文推断

**原子工具**: + resolve_time, suggest_columns

---

### 阶段3: 聚合分析 ✅
**一句话**: 跨行汇总统计——分组排名、时间趋势、统计指标。

**能力边界**: GROUP BY 聚合、排名 TOP N、时间趋势、AVG/SUM/COUNT、多维分组

**原子工具**: + run_aggregation

**关键发现**:
- 数据库有大量预聚合列（累计值、同比率），多数统计问题无需 GROUP BY
- 排名查询直接 ORDER BY 金额列即可（每行已是一个地区的数据）
- 真正需要 GROUP BY 的是跨月汇总各市、跨项目汇总等场景

---

### 阶段4: 跨表分析 ✅
**一句话**: 跨预算类型对比分析——并行查询+LLM综合对比、同构表 UNION ALL 合并、字典表 JOIN 关联、表关系智能发现。

**原子工具**: + find_relations, union_query（共10个工具）

**技能**: cross_table_analysis.md（4种跨表策略）

---

### 阶段5: 深度洞察 ✅
**一句话**: 子查询、窗口函数、异常检测——从"看懂数据"到"发现洞察"。

**原子工具**: + run_subquery, calc_ratio, detect_anomalies（共13个工具）

**技能**: advanced_analysis.md（子查询/占比/环比LAG/异常检测4种模式）

---

### 阶段6: 会话记忆 ✅
**一句话**: 跨会话上下文持久化——关浏览器重开，Agent记得上次查了什么。

**实现方式**: 服务端 sessions/current.json 存上下文 + history.json 存50条查询历史。不需要新原子工具。

**技能**: session_memory.md（复用策略/优先级规则）

---

### 阶段7: 智能容错 📋（最终阶段）

**产品价值**: SQL 报错时 Agent 自动诊断原因并透明修正，用户永远看不到技术错误信息。从"报错了找管理员"到"自动修复无感知"。

**核心区分**: 阶段1-6解决了"能查什么、怎么查、查完怎么办"，阶段7解决"查错了怎么办"。这是产品质量的最后一公里——让非技术用户永远不会被 SQL 错误吓到。

#### 为什么需要阶段7

当前用户在查询时可能遇到：
- "column not found" → 满屏英文错误，完全看不懂
- 查询超时 → 卡住不动，不知道要等还是刷新
- ROWNUM 语法错误 → Agent 知道怎么写但偶尔写错位置
- 表名不存在 → Agent 选错了表，用户不知道怎么办

而这些错误的修复方式是 LLM **有能力做到的**——describe_table 重确认列名、search_schema 重定位表、调整 ROWNUM 位置——只是现在没有自动重试机制。

#### 真实用户场景

| 错误场景 | 现在的体验 | 阶段7的体验 |
|---------|-----------|-----------|
| 用户问"收入多少"，Agent 写错列名 | 显示 "ORA-00904: invalid identifier" | Agent 自动 describe_table 找到正确列名 → 修正 SQL → 透明重试 → 用户看到结果 |
| 复杂查询超时 | 卡住20秒后显示超时 | Agent 检测超时 → 自动加 ROWNUM 限制 → 重试 → 告知"已优化查询范围" |
| ROWNUM 位置错误 | SQL 语法错误 | Agent 自动调整为正确的子查询嵌套 → 重试 |
| 表名写错 | "table not found" | Agent search_schema 重新定位 → 替换正确表名 → 重试 |
| 数据类型不匹配 | "type mismatch" | Agent 自动加 TO_NUMBER/TO_CHAR 转换 → 重试 |

#### 功能7.1: 错误自动分类

Agent 捕获 SQL 执行错误后，自动归类：

| 错误类型 | 识别关键字 | 修复策略 |
|---------|-----------|---------|
| 列名不存在 | "invalid identifier"、"column"、"not found" | describe_table 重确认 → 模糊匹配最相似列名 |
| 表名不存在 | "table or view does not exist" | search_schema 重定位 → 替换 |
| 语法错误 | "syntax"、"missing keyword" | 检查达梦特有语法（ROWNUM位置、||拼接）→ 修正 |
| 超时 | "timeout"、"canceled" | 自动加 ROWNUM <= 100 限制 → 重试 |
| 类型错误 | "type mismatch"、"cannot convert" | 自动加转换函数 |

#### 功能7.2: 透明重试机制

当 SQL 执行失败时，Agent 将错误信息作为新一轮的输入，自动调用修正工具，重新生成 SQL 并执行。

**重试流程**:
```
run_sql 失败 → 错误信息 + 原SQL → describe_table/search_schema 修正 → 新SQL → run_sql 重试
```

**约束**:
- 最多重试 2 次（避免死循环）
- 2 次后仍失败 → 告知用户"查询遇到技术困难，建议简化问题" + 展示最后一次尝试的 SQL
- 重试过程对用户透明（用户只看到最终成功结果或友好提示）

#### 功能7.3: 友好错误降级

当自动修复也失败时，不展示原始错误信息，而是给用户友好的中文提示：

| 原始错误 | 用户看到 |
|---------|---------|
| ORA-00904: "XXX" invalid identifier | "未找到'XXX'列，已尝试匹配相似列名但仍失败。您可以换个说法描述查询条件。" |
| ORA-00942: table or view does not exist | "未找到对应的数据表，可能是查询范围不匹配。建议从'一般公共预算'等常用类型开始。" |
| timeout | "查询超时，数据量可能较大。已自动缩小查询范围，或建议指定更具体的地区和时间。" |

#### 新增原子工具: 无需新增

阶段7主要是修改 handler.py 中 `do_run_sql` 的错误处理逻辑——捕获错误后注入修正 prompt，让 Agent 在下一轮自动修正。不需要新的数据库工具。

#### 验收标准
- 列名写错 → 自动修正并重试成功
- 表名不存在 → 自动定位正确表名
- ROWNUM 语法错误 → 自动调整
- 超时 → 自动加限制重试
- 2次重试仍失败 → 友好中文提示，不显示原始错误
- 整个修正过程对用户透明

## 质量基线（所有阶段通用）

1. **不编造**: 所有数据来自工具返回，禁止编造表名/列名/数值
2. **工具优先**: 用户消息注入强制指令，确保每次查询都走工具链
3. **达梦语法正确**: ROWNUM（非 LIMIT），||（非 CONCAT），ORDER BY + ROWNUM 嵌套子查询
4. **不 SELECT ***: 按意图选列
5. **流式输出**: LLM 响应逐 token 流式展示，完成后渲染 markdown
6. **口语化优先**: 所有用户界面和测试问题使用真实口语，不暴露英文表名/列名
