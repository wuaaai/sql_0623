# 阶段8: 查询轨迹积累与复用 — "越用越聪明"

## 产品价值

每次成功的 SQL 查询都是宝贵的知识资产。当前系统每次查询都从零开始——探索表结构、确认列名、试写SQL。如果把成功查询的轨迹积累下来，新问题来时先检索相似模式作为参考，系统就会越来越快、越来越准。

**一句话**: 从"每次重新思考"到"从经验中学习"。

---

## 核心理念

```
用户提问 → 先搜索历史成功模式 → 找到相似的？
  ├─ 找到 → 参考历史模式改写SQL（快、准）
  └─ 未找到 → 正常探索查询 → 成功后保存为新模式（积累）
```

每次成功都是对系统的训练。用得越多，积累越多，回答越快越准。

---

## 功能拆解

### 功能8.1: 成功查询自动收集

每次 `run_sql` 成功（非0行），自动将本次查询保存为一条"查询模式"。

**保存内容（结构化）**:

```json
{
  "id": "uuid",
  "user_question": "河北省2025年1月一般公共预算收入合计",
  "table": "RDYS_LD_YSSC_YSZX_QSYBGGYSSRWC",
  "columns": ["RG_NAME", "YEAR_MONTH", "XM_NAME", "BYS_JE", "BYS_TBB"],
  "filters": {
    "region": "河北省",
    "time": "202501",
    "project": "合计"
  },
  "aggregation": null,
  "sql": "SELECT RG_NAME, YEAR_MONTH, XM_NAME, BYS_JE, BYS_TBB FROM ...",
  "budget_type": "一般公共预算",
  "query_type": "精准筛选",
  "success": true,
  "timestamp": "2026-06-24 10:30",
  "use_count": 1
}
```

**提取维度**: 预算类型、查询类型（精准筛选/聚合排名/时间趋势/同比对比/异常检测）、表名、列名、WHERE条件关键词

### 功能8.2: 查询模式结构化存储

按维度索引存储，方便快速检索。

**存储结构**:
```
query_patterns/
├── index.json              # 总索引（按预算类型+查询类型分组）
├── patterns/
│   ├── p_001.json          # 单个查询模式
│   ├── p_002.json
│   └── ...
└── stats.json              # 统计：热门表、常用列、高频条件
```

**索引维度**:
- 预算类型: 一般公共预算 / 社保 / 国资 / 政府性基金
- 查询类型: 精准筛选 / 聚合排名 / 时间趋势 / 同比对比 / 异常检测 / 占比分析
- 关键词: 从用户问题中提取的业务关键词

### 功能8.3: 相似查询检索

新问题来时，先检索历史成功模式。

**检索策略**（由简到繁）:
1. **关键词匹配**: 用户问题中的关键词（预算类型、地区名、时间词）在历史模式中匹配
2. **表名匹配**: 如果用户提到了特定预算类型，匹配同类型的历史查询
3. **结构相似度**: 比较新问题和历史问题的列/条件重叠度

**匹配结果排序**: 匹配度 × 使用次数 × 时间衰减（越近权重越高）

### 功能8.4: 查询模式复用

找到相似模式后，注入LLM上下文作为参考。

**复用方式**:
```
[参考-相似历史查询]
用户曾问: "河北省2024年1月一般公共预算收入合计"
使用表: RDYS_LD_YSSC_YSZX_QSYBGGYSSRWC
SQL: SELECT RG_NAME, YEAR_MONTH, XM_NAME, BYS_JE, BYS_TBB FROM ... WHERE ...
你的新问题类似，参考以上模式改写即可，不需要重新探索表结构。
```

LLM不需要从零开始探索——知道用哪张表、选哪些列、怎么写WHERE条件。

## 技术方案

### 新增原子工具: `search_patterns`

| 参数 | 说明 |
|------|------|
| `keyword` | 用户问题关键词 |
| `budget_type` | 预算类型（可选） |
| `limit` | 返回最相似的N条，默认3 |

**输出**: 最相似的3-5条历史成功查询模式（含SQL模板）

### 增强现有工具

- `run_sql` 成功后 → 自动调用 `save_pattern` 保存查询模式
- `search_schema` 调用前 → 先调用 `search_patterns`，找到相似模式时跳过探索直接复用

### 新增技能: `pattern_reuse.md`

- 何时复用历史模式 vs 重新探索
- 如何根据相似模式改写SQL（替换参数值）
- 模式的时效性管理

---

## 数据流

```
用户提问 "河北省2026年一般公共预算收入"
  → search_patterns(keyword="河北省 一般公共预算 收入")
  → 找到: "河北省2025年1月一般公共预算收入合计" (相似度92%)
  → 注入参考模式到system_prompt
  → LLM参考模式改写: 时间从2025年1月改为2026年全年
  → run_sql成功
  → save_pattern保存本次查询（模式库+1）
```

## 实现步骤

### 第1步: 查询模式收集与存储 (基础)

- 创建 `tools/pattern_store.py`（save_pattern 函数）
- 修改 `handler.py` 的 `do_run_sql` 成功分支 → 自动调用 save_pattern
- 存储为 `query_patterns/` 目录下的 JSON 文件

### 第2步: 相似查询检索

- 实现 `search_patterns` 工具
- 关键词匹配 + 预算类型匹配 + 简单相似度排序
- 注册到 tools/schema.py

### 第3步: 上下文注入与复用

- 修改 `server.py` → chat端点调用 search_patterns → 注入参考模式到 system_prompt
- LLM 看到参考模式后，优先复用而非重新探索

### 第4步: 模式管理与统计

- 统计仪表：最常用的表、最高频的查询模式
- 模式过期管理（表结构变更时标记旧模式不可用）
- 模式合并（相同查询自动合并，use_count+1）

## 验收标准

| # | 场景 | 预期 |
|---|------|------|
| 1 | 首次查询"河北省一般公共预算收入" | 正常探索→成功→保存模式 |
| 2 | 第二次问"石家庄一般公共预算收入" | search_patterns找到步骤1的模式→复用表名和列名→只改地区条件→更快完成 |
| 3 | 查询10次后查看模式库 | 至少有5+条成功模式（非重复） |
| 4 | 同样的问题问第3次 | 几乎不需要探索，直接复用SQL模板 |
| 5 | 系统重启后 | 模式库持久化存在，继续可用 |
