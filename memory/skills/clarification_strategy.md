# 追问策略设计

## 核心原则（优先级从高到低）

1. **上下文推断优先** — 能从对话历史推断的条件绝不追问
2. **工具自动补全** — resolve_time 自动补时间，search_schema 自动映射预算类型
3. **合理默认兜底** — 推断不出时用默认值，而非追问
4. **追问是最后手段** — 只有 P0 条件（预算类型）无法推断时才追问
5. **最多连续追问2次** — 第3轮强制自动执行

## 信息补全决策树

```
用户输入问题
  ├─ 时间明确？ → 用 resolve_time(用户表达)
  │   └─ 时间缺失？ → resolve_time("") 自动取最新 → 不追问
  ├─ 预算类型明确？（用户说了"一般公共预算/社保/国资/政府性基金"）
  │   ├─ 明确 → search_schema 定位表 → 不追问
  │   └─ 可推断？（上文提到过"河北省一般公共预算"，现在说"社保的呢"）
  │       ├─ 可推断 → 复用上文 → 不追问
  │       └─ 不可推断 → P0 缺失 → 考虑追问
  ├─ 地区明确？（用户说了"河北省/石家庄/XX市"）
  │   ├─ 明确 → WHERE RG_NAME = ... → 不追问
  │   └─ 缺失？ → 默认不筛选 → 不追问（P1 条件可默认）
  └─ 项目明确？
      ├─ 明确 → WHERE XM_NAME LIKE ...
      └─ 缺失？ → 默认"合计" → 不追问（P1 条件可默认）
```

**结论：绝大多数情况下不需要追问。只有预算类型完全无法推断时才需要。**

## 条件优先级

| 级别 | 条件 | 处理策略 |
|------|------|---------|
| P0 | 预算类型、查询意图 | 必须确认。可推断("社保"→SHBXJJ)，但裸"预算"必须问 |
| P1 | 地区、时间、项目 | 可默认：地区→全部、时间→最新、项目→合计 |
| P2 | 行数限制、排序 | 永不问。默认20行，自然顺序 |

## 两轮降级路径

### 第1轮追问（clarify_count=0）
- 问 P0 缺口（预算类型不可推断时、查询意图模糊时）
- P0 已清晰则批量问 P1 缺口
- 话术示例："您关注哪类预算？一般公共预算、社保基金、国有资本，还是政府性基金？"

### 第2轮追问（clarify_count=1）
- 确认剩余 P1 缺口
- 话术示例："好的，[已确认的条件]。剩下两个确认：哪个地区？哪个月份？不指定的话默认查全部地区最新数据。"

### 第3轮自动执行（clarify_count>=2）
- 系统注入强制指令，不再追问
- 所有缺失条件自动填默认值
- 查询结果前加告知："已为您查询[地区范围][预算类型][时间]数据"

## 默认值策略

| 条件 | 默认值 | SQL体现 |
|------|--------|---------|
| 时间 | resolve_time("") 最新数据 | WHERE YEAR_MONTH = '最新' |
| 地区 | 不筛选 | 无 RG_NAME 条件 |
| 项目 | 合计 | WHERE XM_NAME LIKE '%合计%' |
| 预算类型 | 最后兜底：一般公共预算 | 告知用户"默认查询一般公共预算" |
| 查询意图 | 完成情况 | 选 BYS_JE, BYLJS_JE, YSS |

## 追问计数规则

- Agent的回答包含"？"或"?" → 计数+1
- Agent给出了查询结果（不包含"？"）→ 计数归零
- 用户开启新话题（换了完全不同的预算类型或问题类型）→ 计数归零

本文件定义追问的**频率和降级策略**。
